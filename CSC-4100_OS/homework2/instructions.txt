Introduction

For this assignment, you will add cooperative multitasking to your OS.  In particular, you will add processes and an function for switching between processes.

The main driver 

As a first step toward time slicing, you will first implement task switching via a function call.  In a future assignment, you will modify your OS to use a timer to invoke the task switching function.  However, for now you will create a dispatch() function that is callable from your process code that will switch to another process.  

Below is the modified pseudocode for the main() from the previous assignment.  Note the lines with an arrow, which are modifications to the main.  The first arrow line is "call q_init(Ready_q)" so that your ready queue that will order the execution of the processes is initialized properly.  This function, which you will write in C/C++, will set the head and tail pointers of Ready_q to NULL (or zero).  The second arrow line is a call to spawn_process(p1), which will set up an initial context for the process, allocate its PCB, and put a pointer to the PCB in the ready queue.  The argument p1 is a pointer to a function.  This will be the code that the new process will execute.   

The next arrow line after the comments is a call to dispatch_select().  This routine will dequeue the pointer to the process whose PCB is at the head of the Ready_q. You will declare a global variable called Running which is a pointer to a PCB. You will set it to the dequeued pointer.   This Running variable will represent which process currently is running on the processor (which process is in the RUNNING state).    The next line with an arrow is an inlined assembly language instruction (only use inline assembly this once; all you other assembly code should be in an assembly file). This instruction jumps to assembly code that dispatches the process indicated by the variable Running by restoring the process' context and jumping to its first instruction, which completes system startup. Later in this assignment description, you will be shown exactly what actual code you need to have for spawn_process() and dispatch_select()and restore_context.

Ready Queue

You will have to write your own enqueue() and dequeue() functions that will be used by your OS code for the ready queue.  The prototypes should look like the following:

void q_enqueue(PCB_Q_t *q, PCB_t *pcb);
PCB_t *q_dequeue(PCB_Q_t *q);

Follow the above specifications exactly, or you will not get full credit for you implementation.  Likewise, your enqueue and dequeue will be implemented using a linked list structure, not array based. An array based version will be difficult to debug in later assignments.  You should be able to determine the PCB_t struct and PCB_Q_t structs on your own (see spawn_process() code below).  The structure must have a member named sp which is a uint64_t (#include <stdint.h> to get this typedef), followed by an uint32_t member called pid (short for process ID). 

You can use code that you developed in your data structures course, but remember, you cannot call malloc() or new() to create new nodes.  The professor suggest that you add a next pointer inside your PCB_t so that you do not have to wrap the pcb in a node type in order to link it into your queue.  Also remember that you cannot call ANY library routines.  So, if you call cout or printf(), for example, in your queue code, then you will have to remove it, because you cannot call C/C++ library routines.

Figure 2: create_process()


So, what does spawn_process() do in detail?  It sets up a stack for the process and enqueues it on the Ready_q.  You will be doing simple stack based process switching, so the stack starts with initial values for the process's registers (all zeros), including the PSTATE.  Also it pushes the value that control what code is executed when the process is started.

Code that switches between processes

The next thing that you need to do is create a function that can switch between processes. This function will save the context of the current process on the process's stack, enqueue that process, dequeue the next process and make it current, and restore that process's context.  Additionally, you should have a global variable called "Running" that is a pointer to a PCB_t.  This variable is what your OS will use to track which process is currently in the running state. Following is pseudocode for this function:

   ; Note: dispatch must be written in assembly.
    dispatch:
    1)	subtract 16*16 from the stack
    2)	copy all registers to the stack the stack
			(in the order x0,x1,...,x30)
    3)	put the PSTATE into the stack with the
			following code:
			   mrs     x0, SPSR_EL1      // Save PSTATE
			   str     x0, [sp, 16*15+8]
	4)	store the new sp value into the pcb where Running points
			(note that you cannot move sp into memory directly,
             you have to do something like:
             adr     x1, Running
             ldr     x1, [x1]  // put the first value in the PCB (sp) into x1
    		 mov     x4, sp    // We can't str sp directly, so put sp into x4...
    		 str     x4, [x1]  //    Then store the new sp in x4 to the PCB 
	5)	call enqueue passing it address of ReadyQ and the value of
			Running
    restore_context:   // Note that this is where the inline assembly in main jumps
    6)	call dispatch_select()   // The function written in C/C++
    7)  read the stack pointer value out of the pcb at which Running points and put
        	it into the sp register. The code should look
            like the following:
            	adr		x1, Running
                str		x0, [x1]   // store the pointer to the PCB into the Running variable
    			ldr		x2, [x0]   // x2 now has the value of the stack ptr (we cant ldr sp directly)
    			mov		sp, x2     // sp now has the top of stack for the chosen process
    8)	restore the PSTATE stored in the stack
            ldr     x0, [sp, 16*15+8]   // Restore PSTATE
    		msr     SPSR_EL1, x0
    9) copy all register values for x0-x30 from the stack at
            which sp points into those registers.
    10) add 16*16 to sp
    11) return
 
Figure 5: dispatch()

Note that the above code defines the label restore_context that the assembler instruction in the main() will need.  The code after restore_context dequeues the first process off of the ReadyQ, restores its context, and jumps to its code, which is exactly what main() needs to do!
